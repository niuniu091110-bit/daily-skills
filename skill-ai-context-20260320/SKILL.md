# AI Context Window Manager（上下文窗口管理）

## Overview

帮助用户高效管理和优化AI大模型的上下文窗口，解决长对话中的context overflow、记忆丢失、信息混乱等问题。

## Triggers

- "上下文太长了"
- "AI忘了我之前说的"
- "context window满了"
- "对话太长了怎么办"
- "总结一下之前的对话"
- "context overflow"
- "我的prompt太长了"
- "AI记不住之前的内容"
- "长对话优化"
- "怎么给AI更多记忆"

## Core Concepts

### 什么是Context Window

- Context window = LLM一次能处理的Token数量
- 超出会导致：截断、遗忘、性能下降
- 常见限制：4K、8K、32K、128K、200K、1M tokens

### Context污染（Context Pollution）

- 过多无关对话导致重要信息被稀释
- AI在大量历史中迷失重点
- 解决方案：定期摘要 + 清理无关内容

## Actions

### 1. Context窗口分析

当用户遇到context问题时：

```
请提供：
1. 当前对话的大概长度（短/中/长/很长）
2. 核心任务是什么？
3. AI目前在哪方面出了问题（遗忘/混乱/截断）？
```

### 2. Context压缩策略

#### A. 主动摘要法（推荐）

每当对话超过一定轮数，引导用户进行摘要：

```markdown
## 对话摘要模板

**任务背景**：[用户的核心目标]

**已完成**：
- [ ] xxx
- [ ] xxx

**进行中**：
- [ ] xxx

**关键约束/偏好**：
- [ ] xxx

**当前卡点**：
- xxx
```

#### B. 关键信息提取法

将对话中最重要的信息提取为"系统提示"：

```markdown
# 用户上下文（长期记忆）

## 身份
- [身份信息]

## 偏好
- [沟通偏好]
- [技术偏好]

## 当前项目
- [项目描述]
- [技术栈]

## 重要约束
- [必须遵守的规则]
```

#### C. 对话修剪法

删除中间不必要的对话轮次，保留：
- 开头（任务设定）
- 结尾（当前进度）
- 关键决策点

### 3. 不同场景的Context策略

#### 短任务（<10轮）
- 不需要特殊处理
- 保持完整对话

#### 中等任务（10-30轮）
- 每10轮做一次摘要
- 定期清理无关对话

#### 长任务（>30轮）
- 立即建立"项目记忆"文档
- 后续对话在开头引用该文档
- 使用分层：项目上下文 + 当前会话

#### 超长任务（>100轮）
- 分阶段，每个阶段一个独立会话
- 使用外部工具记录状态
- AI只处理当前阶段的问题

### 4. 实用技巧

#### 巧用系统提示词

将关键信息放在系统提示词或每次对话开头：

```
[系统提示]
用户身份：xxx
当前项目：xxx
技术栈：xxx
关键约束：xxx
---

[用户]
帮我解决xxx问题...
```

#### 结构化输出

让AI定期输出结构化状态：

```markdown
## 当前状态总结

**已完成**：...
**进行中**：...
**待解决**：...
**风险点**：...
```

#### 使用外部记忆

将重要信息保存到文件，必要时让AI读取：

```markdown
## 从外部文件读取

请读取 `project_context.md` 获取项目背景。
```

## 常见问题

### Q: AI说"context window快满了"是什么意思？
A: 意味着对话太长了，AI能处理的信息有限。需要摘要或开始新对话。

### Q: 可以无限延长context吗？
A: 不能，LLM有物理限制。更好的做法是：精选信息 + 外部存储 + 分阶段处理。

### Q: 如何判断context是否"污染"了？
A: 如果AI开始：
- 混淆不同任务的内容
- 重复已经说过的话
- 忽略明显的关键信息
→ 说明context已经污染，需要清理

### Q: 多轮对话后AI表现变差？
A: 这是"context drift"问题。解决方法是：
1. 开始新对话
2. 用摘要重建上下文
3. 只携带"必要信息"继续

## Cost

完全免费（使用上述策略即可）

## Limitations

- 策略需要用户主动配合执行
- 不适用于极低context限制的模型
