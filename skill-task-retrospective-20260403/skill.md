# Task Retrospective Logger

## Overview
在任务执行过程中主动记录关键决策点、替代方案、最终选择及结果。任务结束后生成结构化复盘报告，存入 `memory/` 供后续参考。解决 AI "事后忘光"、同类问题重复踩坑的问题。

**核心价值**：把每一次任务执行变成组织的学习资产。

## 触发词
- "记住这个决定"
- "复盘一下"
- "这次做得好/不好"
- "下次可以改进"
- "记录我的决策"
- "为什么选了方案A"
- "这次踩了什么坑"
- "任务结束了，帮我写复盘"

## 决策点记录格式

遇到关键决策时，在 `memory/YYYY-MM-DD.md` 中插入：

```markdown
## [HH:MM] 决策点 #N
**问题**: <简述要解决的问题>
**选项**:
- A: <方案A描述> → <潜在结果>
- B: <方案B描述> → <潜在结果>
- C: <方案C描述> → <潜在结果>
**选择**: <选了什么>
**理由**: <核心判断依据>
**风险**: <已知的潜在风险>
**预期**: <预期结果>
```

## 复盘报告模板

任务结束后生成以下结构：

```markdown
# 任务复盘报告 — YYYY-MM-DD

## 基本信息
- 任务类型: <分类>
- 执行时长: <估算>
- 复杂度: ⚡⚡⚡⚡⚡ (1-5)

## 决策点回顾
列出所有决策点及最终选择

## 成功之处 ✅
<做得好的3个点>

## 教训 🚨
<需要改进的3个点>

## 规律提炼 📌
<抽象成可复用的原则/模式>

## 下次行动计划 🎯
<针对教训的具体改进>

## 相关决策
链接到相关 memory 文件
```

## 使用时机

### 强制记录（每次都要）
- 方案选型（技术栈、架构、设计模式）
- 权衡取舍（性能 vs 可维护性、速度 vs 质量）
- 踩坑记录（错误类型、原因、解决方案）

### 推荐记录
- 有效的 prompt 模式
- 工具使用技巧
- 有效的验证方法

### 可选记录
- 探索性尝试（失败的也值得记）
- 直觉判断（记录下来验证）

## 执行流程

### Step 1: 任务开始
确认任务类型，在 memory 文件中建立任务段落

### Step 2: 决策发生时
实时写入决策点，不等任务结束

### Step 3: 任务完成
自动或手动触发复盘生成

### Step 4: 规律提炼
用 `<theme>` 标签归类到 MEMORY.md

## 示例

### 决策点示例
```markdown
## [14:23] 决策点 #1
**问题**: 用户需要定时提醒功能，用 cron 还是消息队列？
**选项**:
- A: CronJob（轻量、K8s 原生）→ 适合简单定时
- B: Redis Queue + Worker（支持重试、持久化）→ 适合复杂任务
- C: 云事件调度器 → 需外部依赖
**选择**: A
**理由**: 提醒逻辑简单，不需要重试；减少外部依赖
**风险**: 节点重启可能丢任务（可接受）
**预期**: 定时触发，稳定运行
```

### 复盘报告示例
```markdown
# 任务复盘报告 — 2026-04-03

## 基本信息
- 任务类型: 功能开发-提醒系统
- 执行时长: ~45分钟
- 复杂度: ⚡⚡⚡ (3/5)

## 决策点回顾
- #1: 定时方案选型 → 选择 CronJob（正确，简单够用）

## 成功之处 ✅
- 选择了最简单的方案，避免过度设计
- 提前考虑了节点重启场景

## 教训 🚨
- 没有提前准备集成测试用例
- 没有记录 .env 示例变量

## 规律提炼 📌
- "提醒类需求优先 Cron，消息队列留给需要重试的场景"
- "新功能必须有对应的测试或手动验证 checklist"

## 下次行动计划 🎯
- [ ] 功能开发完成后添加基本集成测试
- [ ] 记录所有需要的环境变量到 .env.example
```

## 规则

1. **实时记录**：决策发生时就记，不要靠记忆
2. **简洁具体**：「选了 Cron 因为简单」比「经过深思熟虑」有用
3. **记录失败**：失败决策比成功更有价值
4. **提炼规律**：从个案抽象到可复用原则
5. **行动导向**：每个教训必须有对应改进行动

## Cost
免费，纯文本记录

## Limitations
- 需要人工触发或明确的决策点识别
- 过度记录会导致效率下降，抓重点
- 规律提炼需要事后主动思考
