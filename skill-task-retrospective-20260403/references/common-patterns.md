# 常见决策场景模式库

## 1. 技术选型类

### 框架/库选择
```
**问题**: XX需求，用哪个框架/库？
**选项**:
- A: 成熟稳定、社区大 → 学习成本中等
- B: 新兴轻量、性能好 → 风险未知
- C: 自己造轮子 → 完全可控但费时
**选择**: 
**原则**: 
  - 80%场景选成熟方案
  - 新技术用在非关键路径
  - 造轮子需要>3次重复需求才值得
```

### 架构决策
```
**常见选项**:
- 宏服务 vs 单体
- 同步 vs 异步
- 缓存 vs 直接查
- CDN vs 源站直出

**决策树**:
- 团队<5人 → 优先单体
- 性能要求极高 → 按需拆分
- 变更频繁模块 → 独立部署
- 读多写少 → 加缓存
```

## 2. 问题解决类

### 方案路径
```
**问题**: XX不工作
**选项**:
- A: 立刻试错（快但可能破坏）→ 
- B: 看文档/日志分析（慢但准）→ 
- C: 问人或搜索（利用外部知识）→ 
**选择**: B优先，然后A验证
**原则**: 
  - 陌生问题先理解再动手
  - 熟悉问题可快速试错
  - 破坏性操作前备份
```

### 调试策略
```
**常见选项**:
- 日志分析
- 断点调试
- 二分排除法
- 对比法（正常 vs 异常）

**选择原则**:
- 日志足够 → 看日志
- 需要理解流程 → 断点
- 范围不明 → 二分
- 有参照物 → 对比
```

## 3. 优先级类

### 任务排序
```
**问题**: 同时来了多个任务，先做哪个？
**选项**:
- A: 紧急的先做
- B: 重要的先做
- C: 简单的先做（快速交付）
**选择**: 紧急×重要矩阵
**框架**:
  - 紧急+重要 → 立刻做
  - 重要不紧急 → 计划做
  - 紧急不重要 → 委托做
  - 不紧急不重要 → 忽略做
```

## 4. 设计类

### UI方案选择
```
**问题**: XX功能界面怎么设计？
**选项**:
- A: 简单直接 → 开发快
- B: 复杂炫酷 → 吸引用户但费时
- C: 参考竞品 → 有依据但无特色
**选择原则**:
  - MVP阶段选A
  - 核心流程可投入设计
  - 参考竞品做差异化
```

## 5. 沟通决策类

### 解释详细度
```
**场景**: 需要向非技术人员解释技术决策
**选项**:
- A: 详细技术解释 → 准确但可能听不懂
- B: 类比比喻 → 易懂但可能不准确
- C: 简化结论 → 快速但不透明
**选择**:
  - 对方需要做决定 → B或A
  - 对方只是知情 → C
  - 对方要验证 → A+备好细节
```

## 6. 代码类

### 重构时机
```
**问题**: 要不要现在重构这段代码？
**选项**:
- A: 现在重构 → 影响当前任务
- B: 后续迭代时重构 → 积累技术债
- C: 永远不重构 → 保持稳定
**决策树**:
  - 重构收益 > 成本？→ 重构
  - 不重构会阻止新功能？→ 重构
  - 代码稳定基本不变？→ 不重构
  - 重构风险高？→ 等等
```

### 测试策略
```
**选项**:
- A: 写完整单元测试 → 慢但覆盖全
- B: 只写关键路径测试 → 平衡
- C: 不写测试只验证 → 快但危险
**选择原则**:
  - 核心业务逻辑 → 完整测试
  - 工具类/辅助函数 → 关键路径
  - 临时脚本/一次性的 → 不测试
```

## 7. 风险类

### 风险处理
```
**选项**:
- A: 规避（换个方案）
- B: 减轻（加防护措施）
- C: 接受（准备好承受）
- D: 转移（保险/外包）
**选择树**:
  - 能规避 → A
  - 不能规避但能缓解 → B
  - 发生概率低影响小 → C
  - 概率低影响大 → D
```
